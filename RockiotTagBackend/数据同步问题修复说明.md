# 数据同步问题修复说明

## 问题分析

经过详细分析，发现以下导致数据缺失的关键问题：

### 1. **缓存机制过于保守**（最严重）
- `getDeviceList` 有30秒缓存
- `refreshLocation` 有60秒缓存
- 这导致即使供应商服务器有新数据，你的服务器也返回旧数据

### 2. **设备列表获取不完整**
- 原代码只获取前100个设备：`getDeviceList(1, 100)`
- 如果超过100个设备会遗漏

### 3. **同步频率太低**
- 定时同步间隔是300秒（5分钟）
- 对于实时位置追踪来说太慢了

### 4. **位置历史同步时间范围有限**
- 只同步最近7天或30天的数据
- 如果设备的历史数据超过这个时间范围，就不会被同步

---

## 已实施的修复

### ✅ 修复1：减少缓存时间（VendorApiService.java）
```java
// 原来
private static final long REFRESH_LOCATION_CACHE_TTL = 60 * 1000;  // 60秒
private static final long GET_DEVICE_LIST_CACHE_TTL = 30 * 1000;   // 30秒

// 修改后
private static final long REFRESH_LOCATION_CACHE_TTL = 10 * 1000;  // 10秒
private static final long GET_DEVICE_LIST_CACHE_TTL = 10 * 1000;   // 10秒
```

### ✅ 修复2：增加每页设备数量（VendorApiService.java）
```java
// 原来
int pageSize = 100;

// 修改后
int pageSize = 200;  // 增加每页数量，减少请求次数
```

### ✅ 修复3：使用完整设备列表（DataSyncService.java）
```java
// 原来
List<Map<String, Object>> devices = vendorApiService.getDeviceList(1, 100);

// 修改后
List<Map<String, Object>> devices = vendorApiService.getDeviceListAll();
```

### ✅ 修复4：提高同步频率（application.properties）
```properties
# 原来
sync.interval=300000  # 5分钟

# 建议修改为
sync.interval=60000   # 1分钟
```

### ✅ 修复5：优化历史数据同步策略
- 前台同步：7天（快速响应）
- 后台同步：30天（确保完整性）

## 进一步优化建议

### 1. 添加实时监控接口
创建一个API端点来检查同步状态：
```java
@GetMapping("/api/sync/status")
public Map<String, Object> getSyncStatus() {
    // 返回最后同步时间、设备数量、记录数量等
}
```

### 2. 实现增量同步
只同步自上次同步以来有变化的数据，而不是每次都全量同步。

### 3. 添加数据对比工具
创建一个工具类来对比供应商服务器和你服务器的数据差异：
```java
// 对比两个服务器的设备数量和位置数据
GET /api/sync/compare?deviceNum=xxx
```

### 4. 优化数据库查询
为常用查询添加索引：
```sql
CREATE INDEX idx_user_device ON location_records(user_id, device_num, timestamp);
CREATE INDEX idx_device_timestamp ON device_history(device_num, timestamp);
```

### 5. 考虑WebSocket推送
如果供应商API支持，可以使用WebSocket接收实时位置更新，而不是轮询。

部署后，请监控以下指标：

1. **设备数量对比**：定期对比供应商API返回的设备数和你服务器存储的设备数
2. **位置记录数量**：检查每天新增的位置记录数是否合理
3. **同步延迟**：从设备上报到你服务器存储的时间差
4. **错误率**：同步失败的次数和原因
