### 1. DatabaseHelper 新增方法

**文件**: `app/src/main/java/com/RockiotTag/tag/DatabaseHelper.java`

**新增方法**: `getLatestRecordTimestamp(String deviceId)`

```java
// 获取设备最新的位置记录时间戳
public long getLatestRecordTimestamp(String deviceId) {
    SQLiteDatabase db = this.getReadableDatabase();
    long latestTimestamp = 0;
    
    try {
        Cursor cursor = db.query(TABLE_LOCATION_HISTORY,
            new String[]{"MAX(" + COLUMN_HISTORY_TIMESTAMP + ")"},
            COLUMN_HISTORY_DEVICE_ID + "=?",
            new String[]{deviceId},
            null, null, null);
        
        if (cursor.moveToFirst() && !cursor.isNull(0)) {
            latestTimestamp = cursor.getLong(0);
            Log.d(TAG, "Latest record timestamp for device " + deviceId + ": " + latestTimestamp);
        }
        cursor.close();
    } catch (Exception e) {
        Log.e(TAG, "Error getting latest timestamp: " + e.getMessage());
    } finally {
        db.close();
    }
    
    return latestTimestamp;
}
```

**作用**: 用于查询本地数据库中最新的轨迹点时间戳，支持增量同步策略。

**修改前**:

**修改后**:
```java

}
```

**工作流程**:
```
用户加载轨迹
    ↓
检查本地是否有缓存数据？
    ├─ 否 → 从服务器完整同步
    └─ 是 → 检查数据新鲜度
            ├─ 数据 < 5分钟 → 直接使用本地缓存 ✅
            └─ 数据 ≥ 5分钟 → 增量同步缺失数据
```

**关键代码**:
```java
// 如果本地有数据且不是强制同步，检查数据新鲜度
if (!forceSync && locationRecords != null && !locationRecords.isEmpty()) {
    long newestTimestamp = locationRecords.get(locationRecords.size() - 1).getTimestamp();
    long cacheAge = System.currentTimeMillis() - newestTimestamp;
    
    // 如果数据在 5 分钟内，直接使用本地缓存
    if (cacheAge < 300000) {
        Log.d(TAG, "✅ Using cached data (age: " + (cacheAge / 1000) + "s)");
        renderTrack(locationRecords);
        return;
    } else {
        Log.d(TAG, "Cache age: " + (cacheAge / 1000) + "s, need to sync");
    }
}
```

#### 优化 2: 增量同步策略

**新增方法**: `syncTrackDataIncremental(...)`

**功能**: 
- 只同步本地缺失的数据（从本地最新时间到结束时间）
- 合并已有数据和新数据
- 自动去重和排序

**关键代码**:
```java
// 增量同步策略：检查本地最新数据
if (locationRecords != null && !locationRecords.isEmpty()) {
    long localLatestTime = locationRecords.get(locationRecords.size() - 1).getTimestamp();
    long syncStartTime = Math.max(startTime.getTimeInMillis(), localLatestTime);
    
    // 如果本地数据已经覆盖查询范围，不需要同步
    if (syncStartTime >= endTime.getTimeInMillis()) {
        Log.d(TAG, "✅ Local data is up to date, no sync needed");
        renderTrack(locationRecords);
        return;
    }
    
    // 增量同步缺失的数据
    Log.d(TAG, "🔄 Incremental sync from: " + sdf.format(new java.util.Date(syncStartTime)));
    syncTrackDataIncremental(deviceNum, syncStartTime, endTime.getTimeInMillis(), locationRecords);
}
```

## 🔧 技术细节

### 缓存有效期配置

当前设置为 **5 分钟**（300,000 毫秒）：

```java
if (cacheAge < 300000) { // 5 分钟
    // 使用缓存
}
```

**可调整参数**:
- 更短（如 1 分钟）: 数据更新更及时，但网络请求更多
- 更长（如 10 分钟）: 减少网络请求，但数据可能稍旧

**建议**: 根据实际业务需求调整，对于实时性要求不高的轨迹数据，5-10 分钟是合理范围。

### 数据合并逻辑

增量同步后的数据处理流程：

```java
// 1. 创建合并列表
List<LocationRecord> allRecords = new ArrayList<>();
if (existingRecords != null) {
    allRecords.addAll(existingRecords); // 添加已有数据
}

// 2. 加载新同步的数据
List<LocationRecord> newRecords = databaseHelper.getLocationRecords(...);
if (newRecords != null && !newRecords.isEmpty()) {
    allRecords.addAll(newRecords); // 添加新数据
    
    // 3. 按时间排序
    Collections.sort(allRecords, ...);
    
    // 4. 去重（基于时间戳）
    List<LocationRecord> uniqueRecords = new ArrayList<>();
    if (!allRecords.isEmpty()) {
        uniqueRecords.add(allRecords.get(0));
        for (int i = 1; i < allRecords.size(); i++) {
            if (allRecords.get(i).getTimestamp() != allRecords.get(i - 1).getTimestamp()) {
                uniqueRecords.add(allRecords.get(i));
            }
        }
    }
    
    // 5. 渲染
    renderTrack(uniqueRecords);
}
```

## ⚠️ 注意事项

### 1. 数据库版本兼容性

新增的 `getLatestRecordTimestamp` 方法使用的是标准 SQL 查询，与现有数据库结构完全兼容，无需升级数据库版本。

### 2. 线程安全

- 数据库查询在后台线程执行（`syncTrackDataIncremental` 中）
- UI 更新通过 `runOnUiThread` 确保在主线程执行
- 原有代码已保证线程安全

### 3. 内存管理

- 合并数据时会创建新的 `ArrayList`，注意大数据量时的内存占用
- 建议在 `onDestroy` 中清理缓存数据（原有代码已实现）

### 4. 异常处理

- 网络请求失败时会降级使用本地缓存
- 数据库查询失败会返回空列表，不会导致崩溃
- 所有异常都已捕获并记录日志
