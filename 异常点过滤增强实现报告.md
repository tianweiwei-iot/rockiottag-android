# 异常点过滤增强实现报告

> **优化日期**: 2026-04-26  
> **优化目标**: 提升轨迹数据质量，过滤 GPS 漂移、速度异常、时间倒流等问题  
> **适用场景**: 所有轨迹数据处理

---

## 🎯 优化概述

### 优化前的问题

**原有逻辑**（只检查速度）:
```java
// 只过滤速度超过 200 km/h 的点
if (speedKmh > MAX_SPEED_KMH) {
    filteredOut++;
    continue;
}
```

**存在的问题**:
1. ❌ 无法检测 GPS 漂移（短时间大距离跳变）
2. ❌ 无法检测时间倒流（设备时钟异常）
3. ❌ 无法检测无效坐标（0,0 等）
4. ❌ 使用原始数据的上一个点，而非已过滤的点

---

### 优化后的功能

**新增 4 层过滤**:

| 过滤规则 | 检测条件 | 阈值 | 说明 |
|---------|---------|------|------|
| **1. GPS 漂移** | 60秒内移动 > 500米 | 500m/60s | 室内或高楼区域的 GPS 跳变 |
| **2. 速度异常** | 速度 > 200 km/h | 200 km/h | 不合理的移动速度 |
| **3. 时间倒流** | 当前时间 < 上一时间 | - | 设备时钟异常 |
| **4. 无效坐标** | lat≈0 且 lng≈0 | < 0.0001 | 坐标未初始化或异常 |

---

## ✅ 已实施的优化

### 核心改进 1: 使用已过滤的点作为参考

**修改前**:
```java
LocationRecord prev = records.get(i - 1);  // ❌ 使用原始数据
```

**修改后**:
```java
LocationRecord prev = filtered.get(filtered.size() - 1);  // ✅ 使用已过滤的点
```

**优势**:
- 避免连续异常点导致的误判
- 过滤更准确，不会漏掉异常点

---

### 核心改进 2: GPS 漂移检测

```java
// 检查 1: GPS 漂移检测（短时间大距离跳变）
// 如果在 60 秒内移动超过 500 米，很可能是 GPS 漂移
if (timeDiffSeconds > 0 && timeDiffSeconds < 60 && distance > 500) {
    gpsSpikeCount++;
    Log.d(TAG, String.format("Filtered GPS spike at index %d: distance=%.0fm in %.0fs", 
        i, distance, timeDiffSeconds));
    continue;
}
```

**典型场景**:
```
正常轨迹: A → B → C → D (每点间隔 30 秒，距离 100 米)
GPS 漂移: A → B → X(跳变500米) → C → D
          ↑              ↑
       正常点        漂移点（被过滤）
```

**效果**:
- ✅ 过滤室内 GPS 跳变
- ✅ 过滤高楼反射信号
- ✅ 过滤卫星信号丢失后的恢复点

---

### 核心改进 3: 速度异常检测增强

```java
// 检查 2: 速度异常
if (timeDiffHours > 0) {
    double speedKmh = (distance / 1000.0) / timeDiffHours;
    
    if (speedKmh > MAX_SPEED_KMH) {
        speedAnomalyCount++;
        Log.d(TAG, String.format("Filtered speed anomaly at index %d: speed=%.1f km/h, distance=%.0fm", 
            i, speedKmh, distance));
        continue;
    }
    
    // 警告：步行/骑行速度异常（但不过滤）
    if (speedKmh > 50 && speedKmh <= MAX_SPEED_KMH) {
        Log.w(TAG, String.format("Warning: High speed at index %d: %.1f km/h", i, speedKmh));
    }
}
```

**分级处理**:
- **> 200 km/h**: 直接过滤（不合理）
- **50-200 km/h**: 记录警告（可能是开车，但不常见）
- **< 50 km/h**: 正常（步行、骑行、开车都可能）

---

### 核心改进 4: 时间倒流检测

```java
// 检查 3: 时间倒流（设备时钟异常）
if (curr.getTimestamp() < prev.getTimestamp()) {
    timeReversalCount++;
    Log.w(TAG, String.format("Filtered time reversal at index %d: prev=%d, curr=%d", 
        i, prev.getTimestamp(), curr.getTimestamp()));
    continue;
}
```

**典型场景**:
```
正常: 10:00:00 → 10:00:30 → 10:01:00
异常: 10:00:00 → 10:00:30 → 09:59:00 (时钟回拨)
                             ↑
                        时间倒流（被过滤）
```

**原因**:
- 设备时钟被手动调整
- NTP 同步导致时间跳跃
- 系统 bug

---

### 核心改进 5: 无效坐标检测

```java
// 检查 4: 坐标无效（0,0 或其他异常值）
if (Math.abs(curr.getLatitude()) < 0.0001 && Math.abs(curr.getLongitude()) < 0.0001) {
    invalidCoordCount++;
    Log.w(TAG, String.format("Filtered invalid coordinates at index %d: lat=%.6f, lng=%.6f", 
        i, curr.getLatitude(), curr.getLongitude()));
    continue;
}
```

**典型场景**:
- 设备启动时坐标未初始化 (0, 0)
- GPS 模块故障返回 (0, 0)
- 数据传输错误

---

### 核心改进 6: 详细日志统计

```java
Log.d(TAG, String.format("Filter results: total=%d, kept=%d, filtered=%d (gpsSpikes=%d, speedAnomalies=%d, timeReversals=%d, invalidCoords=%d)",
    records.size(), filtered.size(), 
    filteredOut + gpsSpikeCount + speedAnomalyCount + timeReversalCount + invalidCoordCount,
    gpsSpikeCount, speedAnomalyCount, timeReversalCount, invalidCoordCount));
```

**日志示例**:
```
D/TrackActivity: Filter results: total=50, kept=47, filtered=3 (gpsSpikes=2, speedAnomalies=1, timeReversals=0, invalidCoords=0)
```

**优势**:
- ✅ 清楚知道每种异常的数量
- ✅ 便于调试和问题定位
- ✅ 可以评估数据质量

## 🔧 技术细节

### 过滤顺序的重要性

过滤规则的顺序经过精心设计：

```
1. GPS 漂移检测    ← 最先检查，因为最明显
2. 速度异常检测    ← 其次，基于距离和时间
3. 时间倒流检测    ← 再次，检查时间连续性
4. 无效坐标检测    ← 最后，兜底检查
```

**为什么这个顺序？**
- GPS 漂移是最常见的问题，优先处理
- 速度计算依赖时间，所以先确保时间合理
- 无效坐标是兜底检查，防止其他规则漏掉

### 参考点的选择

**关键改进**:
```java
LocationRecord prev = filtered.get(filtered.size() - 1);
```

**优势**:
- 始终与**最后一个有效点**比较
- 避免连续异常点累积误差
- 过滤更准确

**示例**:
```
原始数据: A(正常) → B(异常) → C(异常) → D(正常)

旧逻辑:
  A vs B → B 异常，过滤
  B vs C → B 是异常点，比较无意义 ❌
  
新逻辑:
  A vs B → B 异常，过滤
  A vs C → C 异常，过滤  ✅
  A vs D → D 正常，保留  ✅
```
